================================================================================
                    WISEVISION — SHOP PEOPLE COUNTING SOLUTION
                          Logic explanation of main1.py
================================================================================

GOAL
--------------------------------------------------------------------------------
From a fixed shop CCTV video, automatically measure:
  - X = how many CLIENTS entered the shop (crossed the door line going in)
  - Y = how many clients the SELLER interacted with
And produce an annotated output video + a final client log table.

The seller is ONE fixed person (behind the counter). Everyone else is a client.


================================================================================
1. THE FULL PIPELINE (per frame)
================================================================================
For each processed frame we run, in order:

  1) YOLO11s detection      -> finds all "person" boxes
  2) ByteTrack              -> gives each detection a temporary track_id
  3) RoleTracker            -> decides SELLER vs CLIENT and gives a STABLE id
  4) Interaction detection  -> seller <-> client proximity (5 s rule)
  5) Line crossing          -> ENTER / EXIT counting (two-line method)
  6) Draw + write the frame

After the whole video:
  7) Post-processing        -> fix wrong exit attributions
  8) Print the client table + summary (X and Y)

DIAGRAM — per-frame pipeline:

   ┌──────────────┐
   │  video frame │
   └──────┬───────┘
          │ every 2nd frame (SKIP=2); others just re-draw last boxes
          v
   ┌──────────────┐   person boxes        ┌──────────────┐
   │   YOLO11s    │ ────────────────────> │  ByteTrack   │  temp track_id
   └──────────────┘                       └──────┬───────┘
                                                 │
                                                 v
                                        ┌──────────────────┐
                                        │   RoleTracker    │  SELLER / CLIENT
                                        │  (stable id =    │  + stable first_pid
                                        │   first_pid)     │
                                        └────────┬─────────┘
                                                 │ list of people
                        ┌────────────────────────┼────────────────────────┐
                        v                         v                        v
                ┌───────────────┐        ┌────────────────┐       ┌───────────────┐
                │ Interaction   │        │ Line crossing  │       │ Draw + write  │
                │ (5 s rule)    │        │ (two lines)    │       │  the frame    │

                        └────────────┬───────────┘

                │ Print table + summary (X and Y)     │
                └────────────────────────────────────┘

FRAME SKIPPING (SKIP = 2):
  We only fully process every 2nd raw frame. The skipped frames just re-draw the
  last known boxes. At 25 fps this means ~12.5 processed frames per second, which
  is enough for tracking while roughly halving the compute.


================================================================================
2. DETECTION (YOLO11s)
================================================================================
  - Model: yolo11s.pt, class 0 = "person".
  - CONF_SELLER = 0.1  : low threshold so even a faint seller box is detected.
  - CONF_CLIENT = 0.20 : clients need a bit more confidence to be created.
  - imgsz = 640        : inference resolution (video is resized internally).

YOLO only says "here are person boxes". It has NO memory between frames — that
is the job of the trackers below.


================================================================================
3. BYTETRACK (temporary ids)
================================================================================
  - ByteTrack matches detections frame-to-frame using motion + box overlap (IoU).
  - It outputs a track_id per detection.
  - It is fast but LOSES the id easily (occlusion, people crossing, missed
    detections). So we never trust it alone — its id is only one hint used by
    the RoleTracker below.
  - Config: track_thresh=0.1, track_buffer=360, match_thresh=0.85.


================================================================================
4. ROLETRACKER (the core of the solution)
================================================================================
This custom class turns raw detections into stable people with a role.

KEY IDEA — "first_pid":
  Every physical person gets a permanent id called first_pid. Even if ByteTrack
  loses them and they get re-detected, RoleTracker tries to give them back their
  ORIGINAL first_pid. All counting is keyed on first_pid, not on ByteTrack id.

The update() runs in 3 steps every frame:

  --- STEP 1: SELLER GETS FIRST PICK ---
  If the seller is locked, he chooses his detection BEFORE any client:
    - IoU-match his last box against all detections, within SELLER_RADIUS px.
    - Reject a detection if a client is closer to it than the seller (so the
      seller does not jump onto a nearby client).
    - If IoU fails -> positional fallback: nearest detection within radius that
      is not too close to a known client.

  --- STEP 2: CLIENTS (IoU + distance) ---
  For the remaining detections (conf >= CONF_CLIENT):
    - Match each detection to an existing client by IoU, but only accept it if
      the centers are also within CLIENT_RADIUS (= 150 px). This avoids matching
      a far-away box that happens to overlap.
    - Clients that still have no match -> positional fallback: nearest existing
      client within CLIENT_RADIUS.

  --- STEP 3: REMAINING DETECTIONS -> NEW PERSON (with re-ID) ---
  A leftover detection is either a brand-new person or a re-appearance of someone
  we lost. Before creating a new id we try 4 re-ID mechanisms, in order, to give
  back the original first_pid:

     1) ByteTrack id lookup : if this ByteTrack id was already linked to a
        first_pid before, reuse it (most reliable signal).
     2) Evicted cache       : a person whose track was fully dropped is kept in a
        cache for a while; if a new detection appears near where they vanished,
        inherit their first_pid.
     3) Ghost client        : a client still "alive" but not seen this frame; if a
        new detection is near their last position, it is the same person -> merge.
     4) Last-resort IoU      : if all the above fail, match by very high box
        overlap (IoU > 0.9) OR very small center distance (< 100 px), against
        both the evicted cache and the ghosts.

  Before creating a new person we also drop DUPLICATE boxes (two YOLO boxes for
  the same body): suppressed if center distance < CLIENT_RADIUS OR IoU > 0.5.

  If none of the re-ID mechanisms fire -> it is a genuinely new client and gets a
  fresh first_pid.

  DIAGRAM — how one detection gets a role + stable id:

      ┌────────────────────┐
      │   one detection    │
      └─────────┬──────────┘
                v
        Is the seller's?  ── yes ──>  assign to SELLER  (STEP 1)
                │ no
                v
      Matches an existing      ── yes ──>  same CLIENT   (STEP 2)
      client (IoU + dist)?                 keep its first_pid
                │ no
                v
            STEP 3 — try to recover an old id, in order:
                │
                ├─ 1) known ByteTrack id?      ── yes ──> reuse first_pid
                ├─ 2) near an evicted person?  ── yes ──> reuse first_pid
                ├─ 3) near a ghost client?     ── yes ──> reuse first_pid (merge)
                ├─ 4) high IoU / very close?   ── yes ──> reuse first_pid
                │
                └─ none matched  ──> NEW CLIENT, fresh first_pid

  SELLER LOCK:
    During the first SELLER_LOCK_WINDOW_SECONDS (90 s), the first person created
    is locked as the SELLER. After that, every new person is a CLIENT.

  --- SELLER RESCUE (end of update) ---
  If Step 1 missed the seller but a CLIENT box overlaps the seller's last box
  (IoU > 0.85) for 3 frames in a row, that box is relabeled back to the seller.

GHOSTS AND EVICTION:
  - A client not seen for a while is kept as a "ghost" (GHOST_TTL_FRAMES = 360
    processed frames) so they can be re-matched.
  - When the ghost finally expires it is moved to the evicted_cache (kept ~300
    frames) so a later re-appearance can still recover the original id.


================================================================================
5. INTERACTION DETECTION (seller <-> client)
================================================================================
Goal: count Y = clients the seller actually served.

  - Distance used = EDGE-TO-EDGE gap between the seller box and the client box
    (not center-to-center), so it is independent of how big the boxes are.
  - A client is "close" if that gap < INTERACTION_DIST (350 px).
  - The closeness must last continuously for INTERACTION_MIN_SECS (5 s). A timer
    starts when they get close; if they move apart, the timer resets.
  - When 5 s of continuous proximity is reached, the interaction is CONFIRMED for
    that client's first_pid (counted only once per client).
  - Only clients that actually entered (have an entry) are eligible in main1.py.

  DIAGRAM — edge-to-edge gap and the 5 s timer:

      seller box        client box
      ┌────────┐        ┌────────┐
      │        │<--gap-->│        │     gap < 350 px  => "close"
      └────────┘        └────────┘

      timeline (client stays close):
        close ───────────────────────────> 5 s reached
        |====|====|====|====|====| ✔ CONFIRMED  (Y += 1, once)

      if the client steps away before 5 s:
        close ──────X (moved away)
        |====|====|       timer RESET to 0


================================================================================
6. LINE CROSSING — THE TWO-LINE METHOD
================================================================================
The door is guarded by TWO parallel lines (LINE1 outer, LINE2 inner), only a few
pixels apart.

  - is_in_side() uses the cross-product sign to tell which side of a line a point
    is on.
  - A real crossing means the person crosses BOTH lines in the same direction
    within DUAL_LINE_SECS (0.3 s). A single-line flicker is ignored as noise.
      * Cross both inward  -> ENTER (X += 1)
      * Cross both outward -> EXIT
  - pending_cross[] remembers the first line a person crossed until the second
    line confirms it; if the second line does not come in time, it resets.

  DIAGRAM — the two-line door:

        OUTSIDE                         INSIDE
                  LINE1   LINE2
                    │      │
        ────────►   │      │   client walking IN
          person    │      │
                    │      │
       crosses LINE1 first, then LINE2 within 0.3 s   => ENTER (X += 1)
       crosses LINE2 first, then LINE1 within 0.3 s   => EXIT

       crosses only ONE line then turns back          => ignored (noise)

  WHY TWO LINES INSTEAD OF ONE:
       A single line gives false counts when a person hovers on the line and the
       box jitters back and forth. Requiring BOTH lines in the same direction
       proves a real walk-through, not a jitter.

  CO-ENTERING (Pass 2):
    When 2-3 clients enter together, one may be first detected ALREADY inside the
    lines (so no crossing event fires for them). Pass 2 catches this: if a client
    appears inside right after a real ENTER happened (within the window) and is
    still tracked, they are counted as a co-entering ENTER.

  RE-ENTRY GUARD:
    A client already marked as exited is not counted again if they reappear.


================================================================================
7. POST-PROCESSING — FIX WRONG EXITS
================================================================================
Because tracking is imperfect, sometimes a client EXITS under a NEW id that has
no matching ENTER (an "orphan exit"), while the real person has an ENTER but no
EXIT (an "orphan entry"). We repair this AFTER the video:

  - Two exits that happen close in time (<= 2 s) likely belong to two clients who
    left together.
  - If one of those exits belongs to a properly-tracked client, and that client's
    ENTRY time is close (<= 2 s) to an orphan entry, then the orphan exit is
    re-assigned to that orphan-entry client.
  This recovers the correct (entry, exit) pairing for clients whose id changed.

  DIAGRAM — repairing a broken id:

     what was recorded:            what really happened:

       CLIENT #10  ENTER ✔           one person:
       CLIENT #10  (no exit)           ENTER as #10
                                       ... id changed mid-visit ...
       CLIENT #13  (no enter)          EXIT  as #13
       CLIENT #13  EXIT ✔

     fix: #13's exit happened with another client's exit at the same time,
          and #10's entry matches that other client's entry timing
          => give #13's EXIT to #10   =>   #10 ENTER + EXIT  (one real client)


================================================================================
8. FINAL OUTPUT
================================================================================
CLIENT LOG TABLE (one row per real client):
  - A "real client" = has BOTH an entry AND an exit, and stayed > 3 s
    (very short pairings are noise and are dropped).
  - Columns: CLIENT id | ENTRY time | EXIT time | DURATION | INTERACTION (YES/NO)
  - "Clients served" = how many of them interacted with the seller.

SUMMARY:
  - Clients entered (X), clients exited, output video path.

ON THE VIDEO ITSELF:
  - SELLER box in red, CLIENT boxes in green with "CLIENT #id".
  - "+1" tag on a client once their 5 s interaction is confirmed.
  - The two door lines and live "IN / OUT / Interactions" counters.


================================================================================
9. THE MAIN PARAMETERS (and what they do)
================================================================================
  SKIP                 = 2     process every 2nd frame
  CONF_SELLER          = 0.1   YOLO threshold (seller can be faint)
  CONF_CLIENT          = 0.20  YOLO threshold to create a client
  SELLER_RADIUS        = 450   how far the seller match can reach (px)
  CLIENT_RADIUS        = 150   how far a client match / merge can reach (px)
  GHOST_TTL_FRAMES     = 360   how long a lost client stays re-matchable
  SELLER_ALONE_SECONDS = 5     lone person on screen -> treated as seller
  INTERACTION_DIST     = 350   edge-to-edge gap for "close to seller" (px)
  INTERACTION (time)   = 5 s   continuous proximity needed to count a service
  DUAL_LINE_SECS       = 0.3   max time between crossing line 1 and line 2
  LINE1 / LINE2        = the two door lines (outer / inner)


================================================================================
10. KNOWN LIMITATIONS
================================================================================
  - Tracking is geometry-based (position + box overlap), with NO appearance
    model. Two people who pass very close or get occluded can swap ids, and a
    person who reappears far from where they vanished can get a new id. This is
    the main source of remaining counting errors.
  - Clients already inside the shop when the video starts have no ENTER event.
  - The overhead/angled camera view makes standard ReID models (BoT-SORT /
    StrongSORT) only partly effective — they are trained mostly on side views.
  - Possible upgrades: appearance ReID fine-tuned on this camera, anchor-based
    seller tracking (already prototyped in last.py), and real-time exit
    re-attribution by last-seen position.
